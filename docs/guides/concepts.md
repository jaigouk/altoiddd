alto is not another AI coding tool. It is the architect that runs before builders start. This page explains the concepts alto enforces and why they matter.

## The problem

AI coding tools are fast. They can generate hundreds of lines of code in seconds. But speed without structure produces:

- **Wrong abstractions** — code organized by technical concerns (controllers, models, utils) instead of business domains
- **Anemic domain models** — data containers with no business logic, all behavior in "service" classes
- **Implicit knowledge** — business rules scattered across the codebase with no glossary or documentation
- **Ad-hoc architecture** — no clear boundaries, everything depends on everything

alto solves this by capturing domain knowledge before any code is written. The output is a structured project that AI coding tools can build within guardrails.

## Domain-Driven Design (DDD)

DDD is a methodology for building software that matches the business domain. alto uses DDD's strategic patterns to structure your project.

### Ubiquitous language

The most important DDD concept. A shared vocabulary where business terms are used identically in conversations, documentation, and code.

If the business says "approve invoice", the code says `ApproveInvoice()` — not `UpdateStatus()` or `SetApproved()`.

alto builds the ubiquitous language glossary during guided discovery (see `docs/DDD.md` in your generated project). Every term in the glossary must appear in your code exactly as defined.

### Bounded contexts

A bounded context is an explicit boundary where domain terms have specific, unambiguous meaning. The same word can mean different things in different contexts:

| Term | In Billing context | In Shipping context |
|------|-------------------|-------------------|
| "Order" | A financial record with line items and totals | A shipment request with weight and destination |

alto identifies bounded contexts during guided discovery and generates a separate directory structure for each one:

```
internal/
├── billing/
│   ├── domain/
│   ├── application/
│   └── infrastructure/
└── shipping/
    ├── domain/
    ├── application/
    └── infrastructure/
```

Each context owns its domain model. Cross-context communication happens through domain events or well-defined integration patterns — never by importing another context's internals.

### Aggregates

An aggregate is a cluster of domain objects treated as a single unit for data changes. Every aggregate has a root that controls access to its contents and enforces business rules (invariants).

Example: an `Invoice` aggregate might contain `LineItem` value objects. The invoice enforces the invariant "total must equal sum of line items". You can't modify a line item directly — you go through the invoice root.

alto designs aggregates for core subdomains during guided discovery and documents their invariants in `docs/DDD.md`.

### Domain events

A domain event records something significant that happened. Events are named in past tense: `InvoiceApproved`, `PaymentReceived`, `OrderShipped`.

Events connect bounded contexts. When billing approves an invoice, it emits `InvoiceApproved`. Shipping listens for that event and starts the fulfillment process. Neither context knows about the other's internals.

### Subdomain classification

Not every part of your system needs the same level of engineering investment. alto classifies subdomains during discovery:

| Classification | Investment | Example |
|---------------|------------|---------|
| **Core** | Full DDD treatment — aggregates, invariants, comprehensive tests, strict fitness functions | Your competitive differentiator |
| **Supporting** | Standard architecture — layers, basic tests, moderate fitness functions | Necessary but not differentiating |
| **Generic** | Minimal — use a library, CRUD, single boundary check | Commodity (auth, email, file storage) |

This classification drives how detailed the generated tickets are, how strict the architecture tests are, and how much domain modeling alto invests in each subdomain.

## TDD (Test-Driven Development)

alto enforces a Red/Green/Refactor cycle in generated tickets:

| Phase | What happens |
|-------|-------------|
| **Red** | Write a failing test that describes the expected behavior |
| **Green** | Write the minimum code to make the test pass |
| **Refactor** | Clean up the code while keeping tests green |

Tests describe behavior, not implementation. Test names follow the pattern `TestSubject_WhenCondition_ExpectOutcome`.

Every core subdomain ticket generated by alto includes TDD phases in its acceptance criteria.

## SOLID principles

alto generates code and tickets that follow SOLID:

| Principle | In practice |
|-----------|------------|
| **Single Responsibility** | One struct does one job. One handler per use case. |
| **Open/Closed** | Extend behavior by adding new adapters, not modifying existing ones. |
| **Liskov Substitution** | Every adapter correctly implements its port interface. |
| **Interface Segregation** | Small, focused interfaces — one per concern. |
| **Dependency Inversion** | Handlers depend on port interfaces, never on concrete adapters. |

The key pattern: **ports and adapters**. Business logic defines port interfaces (what it needs). Infrastructure provides adapter implementations (how it's done). The domain never depends on frameworks, databases, or HTTP.

## CQRS-lite

alto separates commands (writes that change state) from queries (reads that return data):

- **Command handlers** live in `application/commands/` — they mutate state and return errors
- **Query handlers** live in `application/queries/` — they return data with no side effects

This is "lite" CQRS — not full event sourcing. State is mutable. Events are notifications, not the source of truth.

## Quality gates

alto generates quality gate configuration that AI agents must pass before completing work:

```bash
go build ./...           # Compile check
go vet ./...             # Static analysis
golangci-lint run ./...  # Meta-linter (errcheck, wrapcheck, etc.)
go test ./... -race      # Tests with race detector
```

All four must pass with zero errors. alto's `check` command runs them all:

```bash
alto check
```

## Architecture fitness functions

Fitness functions are automated tests that enforce architectural rules. alto generates them from your bounded context map using arch-go:

- **Layer rules** — domain layer cannot import application or infrastructure
- **Forbidden dependencies** — one context cannot import another's internals
- **Independence checks** — core subdomains have no coupling to supporting/generic
- **Acyclic sibling checks** — contexts within the same parent don't form cycles

The strictness of these rules depends on subdomain classification. Core subdomains get all four check types. Generic subdomains get a single boundary check.

## Putting it all together

The alto workflow:

```
Idea (README)
  → Guided discovery (10 questions)
  → Domain model (bounded contexts, aggregates, ubiquitous language)
  → Architecture (layers, fitness functions)
  → Tickets (dependency-ordered, TDD phases, complexity-budget-driven detail)
  → AI coding tool builds within guardrails
```

Every step adds structure that prevents the next step from going wrong. By the time an AI coding tool starts writing code, it knows the domain vocabulary, respects context boundaries, follows TDD, and passes architecture tests automatically.
