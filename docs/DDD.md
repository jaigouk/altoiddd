# Domain-Driven Design Artifacts: alty

> **Purpose:** This document captures domain knowledge BEFORE writing architecture
> or code. It ensures we build the right thing by understanding the business domain
> first, using Domain Storytelling and DDD strategic patterns.
>
> **Method:** Generated by applying the 10-question dual-register DDD framework
> (docs/research/20260222_guided_ddd_question_framework_consolidated.md) to
> alty's own PRD and README. Persona: Developer (technical register).

## 1. Domain Stories

### Story 1: New Project Bootstrap (Primary Flow)

**Actors:** User (any persona: Solo Developer, Team Lead, AI Tool Switcher, Product Owner, Domain Expert), alty CLI, AI Coding Tool
**Trigger:** User has a 4-5 sentence project idea

```
1. User writes a project idea in README (4-5 sentences)
2. User runs `alty init`
3. alty CLI detects installed AI tools and global configs
4. alty CLI shows preview of what will be created
5. User confirms the preview
6. alty CLI asks persona detection question ("which best describes you?")
7. alty CLI selects register (technical for developers, non-technical for PO/Domain Expert)
8. alty CLI asks guided DDD questions (10 questions, 5 phases, in selected register)
9. User answers each question in their own language
10. alty CLI plays back understanding after every 3-4 questions
11. User confirms or corrects the playback
12. alty CLI generates PRD from answers
13. alty CLI generates DDD artifacts (domain stories, bounded contexts, ubiquitous language)
14. alty CLI classifies subdomains using complexity budget (Core/Supporting/Generic)
15. alty CLI generates architecture doc informed by DDD and classification
16. alty CLI generates fitness function tests (import-linter + pytestarch) from bounded context map
17. alty CLI generates dependency-ordered beads tickets from DDD artifacts
18. alty CLI previews generated tickets for approval
19. User approves generated tickets
20. alty CLI generates tool-native configs for User's AI Coding Tool
21. User (or their developer) hands project to AI Coding Tool — it builds within guardrails
```

**Key observations:**
- "Preview before action" is a hard constraint — appears at steps 4, 18
- The question flow is the heart of the process — steps 6-11 are where domain knowledge is captured
- Generation is a pipeline: answers → DDD artifacts → architecture → fitness functions → tickets → configs
- Non-technical users (Product Owner, Domain Expert) never see DDD jargon — register adapts automatically
- For PO/Domain Expert: steps 12-21 produce handoff artifacts for a developer to execute

### Story 2: Existing Project Adoption (Rescue Flow)

**Actors:** Team Lead, alty CLI, Existing Codebase
**Trigger:** Team Lead has a messy existing project

```
1. Team Lead runs `alty init --existing` in the project directory
2. alty CLI checks for clean git working tree (abort if dirty)
3. alty CLI creates a new branch `alty/init`
4. alty CLI scans Existing Codebase (code, docs, configs, folder structure)
5. alty CLI performs gap analysis (what's missing vs fully-seeded project)
6. alty CLI shows gap report and proposed changes
7. Team Lead reviews the gap report
8. alty CLI asks clarifying DDD questions about existing domain
9. Team Lead answers questions (bounded contexts, ubiquitous language)
10. alty CLI generates missing artifacts (PRD, DDD, Architecture stubs)
11. alty CLI adapts agent profiles to existing domain language
12. alty CLI runs existing test suite — hard gate: zero regressions
13. If tests pass: Team Lead reviews branch diff, merges when satisfied
14. If tests fail: alty CLI rolls back all changes, reports what broke
```

**Key observations:**
- Branch safety is paramount — never touches main
- Gap analysis is the differentiator vs just scaffolding on top
- Test regression gate is a hard constraint (step 12-14)

### Story 3: Ticket Freshness & Ripple Review

**Actors:** Developer, alty CLI, Beads (issue tracker)
**Trigger:** Developer closes a ticket

```
1. Developer closes a ticket via `bd close <id>`
2. alty CLI traverses dependency graph (dependents + siblings in same epic)
3. alty CLI records context summary of what the closed ticket produced
4. alty CLI flags each affected open ticket with `review_needed` label
5. alty CLI adds context diff comment to each flagged ticket
6. alty CLI shows summary: "Flagged N tickets for review"
7. Later: Developer claims a flagged ticket (`bd update <id> --status in_progress`)
8. AI Agent sees `review_needed` flag and reads context diff
9. AI Agent presents suggested updates to Developer
10. Developer approves, modifies, or dismisses suggestions
11. Flag is cleared, `last_reviewed` is updated
```

**Key observations:**
- Event-driven staleness (not time-based) — this is the competitive differentiator
- Human-in-the-loop: system flags and suggests, human decides
- The context diff is what makes the review actionable (not just "something changed")

### Story 4: Fitness Function Generation

**Actors:** alty CLI, Bounded Context Map, Test Suite
**Trigger:** DDD artifacts completed (bounded context map + subdomain classification)

```
1. alty CLI reads bounded context map from DDD artifacts
2. alty CLI reads subdomain classification (complexity budget)
3. For each bounded context, alty CLI selects contract strictness based on classification:
   - Core: layers + forbidden + independence + acyclic_siblings
   - Supporting: layers + basic forbidden
   - Generic: single forbidden (ACL boundary only)
4. alty CLI generates import-linter TOML contracts in pyproject.toml
5. alty CLI generates pytestarch test files per bounded context
6. alty CLI previews generated tests for approval
7. Developer approves generated tests
8. Tests run as part of quality gates (ruff + mypy + pytest + fitness functions)
```

**Key observations:**
- Classification drives strictness — not all contexts get the same treatment
- Two complementary tools: import-linter (config-based) + pytestarch (code-based)
- Preview before creation — consistent with all generation flows

### Story 5: Domain Story to Ticket Pipeline

**Actors:** alty CLI, DDD Artifacts, Beads
**Trigger:** DDD artifacts completed

```
1. alty CLI reads DDD artifacts (bounded contexts, aggregates, domain events, invariants)
2. alty CLI reads subdomain classification (complexity budget)
3. alty CLI generates one beads epic per bounded context
4. alty CLI generates tasks ordered by aggregate dependency (foundation first)
5. For Core subdomains: tasks include full AC, TDD phases, SOLID mapping, edge cases
6. For Supporting subdomains: tasks include standard AC, basic tests
7. For Generic subdomains: tasks are stubs ("integrate X", verify boundary)
8. alty CLI sets blocking relationships from context map dependencies
9. alty CLI previews generated tickets for approval
10. Developer approves, modifies, or dismisses tickets
11. Approved tickets are created in Beads
```

**Key observations:**
- Two-tier generation: near-term tickets get full detail, far-term are stubs
- Complexity budget determines ticket detail level
- Human approves before creation — no automatic ticket creation

### Story 6: Multi-Tool Config Generation

**Actors:** Developer, alty CLI, AI Coding Tool
**Trigger:** DDD artifacts + architecture completed

```
1. Developer selects target AI tool(s) during bootstrap
2. alty CLI reads domain model (ubiquitous language, bounded contexts, agent personas)
3. alty CLI translates domain model into tool-native config format:
   - Claude Code: .claude/CLAUDE.md, .claude/agents/*.md, .claude/commands/*.md
   - Cursor: .cursor/rules, .cursor/agents
   - Antigravity: TBD (spike needed)
   - OpenCode: TBD (spike needed)
4. Configs contain: ubiquitous language, bounded context rules, quality gates, agent personas
5. alty CLI previews generated configs
6. Developer approves generated configs
7. AI Coding Tool reads configs and builds within guardrails
```

**Key observations:**
- One domain model, multiple output formats — classic adapter pattern
- Tool knowledge must be versioned (conventions change)

## 3. Subdomain Classification

### Classification Flow (Khononov Decision Tree)

Applied to alty's own subdomains:

**Guided Discovery:**
- Could you buy it? → NO (no product does AI-guided DDD questions with dual register + persona detection)
- Complex rules? → YES (10-question framework, branching logic, persona detection, playback loops, artifact extraction)
- If a competitor copied it exactly, would that threaten the business? → YES
- → **CORE**

**Architecture Testing (Fitness Functions):**
- Could you buy it? → NO (Python has no DDD-to-fitness-function pipeline; Java has ArchUnit but nothing equivalent)
- Complex rules? → YES (classification-driven contract selection, bounded context map → test generation pipeline)
- If a competitor copied it exactly, would that threaten the business? → YES
- → **CORE**

**Ticket Pipeline:**
- Could you buy it? → NO (no tool generates DDD-informed, dependency-ordered tickets with TDD phases)
- Complex rules? → YES (aggregate dependency ordering, two-tier generation, complexity-budget-driven detail levels)
- If a competitor copied it exactly, would that threaten the business? → YES
- → **CORE**

**Ticket Freshness & Ripple Review:**
- Could you buy it? → NO (no tool does event-driven staleness detection)
- Complex rules? → YES (dependency graph traversal, context diff capture, flag lifecycle)
- If a competitor copied it exactly, would that threaten the business? → YES
- → **CORE**

**Tool Translation (Multi-Tool Config Generation):**
- Could you buy it? → NO (no tool generates domain-aware configs for multiple AI tools from a single model)
- Complex rules? → MODERATE (adapter pattern per tool, but the mapping logic is straightforward once the domain model exists)
- If a competitor copied it exactly, would that threaten the business? → PARTIALLY (the config formats change often, but the translation itself isn't the core value)
- → **SUPPORTING**

**Knowledge Base:**
- Could you buy it? → PARTIALLY (Context7 exists for library docs, but not for DDD patterns + tool conventions combined)
- Complex rules? → MODERATE (RLM addressing, version tracking, drift detection)
- If a competitor copied it exactly, would that threaten the business? → NO (the content is curated reference material)
- → **SUPPORTING**

**Rescue Mode (Gap Analysis):**
- Could you buy it? → NO (no tool scans Python projects for DDD structure gaps)
- Complex rules? → MODERATE (file scanning, comparison against reference structure, report generation)
- If a competitor copied it exactly, would that threaten the business? → PARTIALLY
- → **SUPPORTING** (P1 — deferred to later phase; basic overlay is scaffolding, smart migration is the hard part)

**File Generation & Template Rendering:**
- Could you buy it? → YES (Jinja2, cookiecutter — commodity)
- Complex rules? → NO (copy files, fill placeholders)
- If a competitor copied it exactly, would that threaten the business? → NO
- → **GENERIC**

**CLI Framework:**
- Could you buy it? → YES (click, typer — commodity)
- Complex rules? → NO (argument parsing, subcommands)
- If a competitor copied it exactly, would that threaten the business? → NO
- → **GENERIC**

**MCP Server Framework:**
- Could you buy it? → YES (fastmcp, mcp-python — commodity)
- Complex rules? → NO (tool registration, request routing)
- If a competitor copied it exactly, would that threaten the business? → NO
- → **GENERIC**

## 4. Bounded Contexts

### Context: Guided Discovery

**Responsibility:** Owns the conversational DDD question flow — from persona detection through 10 questions to artifact extraction. This is the entry point for all new projects.

**Key domain objects:**
- `DiscoverySession` (Aggregate) — tracks the state of a guided discovery conversation
- `Persona` (Value Object) — the detected user persona (Solo Developer, Team Lead, AI Tool Switcher, Product Owner, Domain Expert)
- `Register` (Value Object) — technical or non-technical language level
- `QuestionPhase` (Value Object) — one of 5 phases (Seed, Actors, Story, Events, Boundaries)
- `Question` (Entity) — a dual-register question with technical and non-technical variants
- `Answer` (Value Object) — the user's response to a question
- `Playback` (Value Object) — a summarization of answers for user confirmation
- `DiscoveryCompleted` (Domain Event) — emitted when all questions are answered and confirmed

**External dependencies:** None (pure domain logic). Outputs feed into Domain Model context.

### Context: Domain Model

**Responsibility:** Owns the DDD artifacts produced by Guided Discovery — domain stories, ubiquitous language, bounded contexts, aggregates, events, commands, policies. This is the canonical representation of the user's domain.

**Key domain objects:**
- `DomainModel` (Aggregate) — the complete set of DDD artifacts for a project
- `DomainStory` (Entity) — a "[Actor] [verb] [work object]" narrative
- `UbiquitousLanguage` (Entity) — the glossary of domain terms
- `BoundedContextMap` (Entity) — the contexts and their relationships
- `BoundedContext` (Value Object) — a named context with responsibility and key objects
- `ContextRelationship` (Value Object) — upstream/downstream relationship with integration pattern
- `AggregateDesign` (Entity) — root, contents, invariants, commands, events
- `SubdomainClassification` (Value Object) — Core/Supporting/Generic with rationale
- `DomainModelGenerated` (Domain Event) — emitted when all DDD artifacts are complete

**External dependencies:** Receives input from Guided Discovery (DiscoveryCompleted event).

### Context: Architecture Testing

**Responsibility:** Owns the generation of executable fitness function tests from the bounded context map and subdomain classification. Maps DDD concepts to import-linter contracts and pytestarch rules.

**Key domain objects:**
- `FitnessTestSuite` (Aggregate) — the complete set of generated architecture tests for a project
- `Contract` (Entity) — an import-linter contract (layers, forbidden, independence, acyclic_siblings)
- `ArchRule` (Entity) — a pytestarch rule (layer purity, dependency direction)
- `ContractStrictness` (Value Object) — strict/moderate/minimal based on subdomain classification
- `FitnessTestsGenerated` (Domain Event) — emitted when tests are ready for preview

**External dependencies:** Reads BoundedContextMap and SubdomainClassification from Domain Model context. Writes to File Generation context (generic — file output).

### Context: Ticket Pipeline

**Responsibility:** Owns the generation of dependency-ordered beads tickets from DDD artifacts. Applies two-tier generation and complexity-budget-driven detail levels.

**Key domain objects:**
- `TicketPlan` (Aggregate) — the complete set of generated tickets for a project, before approval
- `GeneratedEpic` (Entity) — one epic per bounded context
- `GeneratedTicket` (Entity) — a task with AC, TDD phases, SOLID mapping, dependencies
- `TicketDetailLevel` (Value Object) — full/standard/stub based on subdomain classification
- `DependencyOrder` (Value Object) — the build sequence derived from aggregate dependencies
- `TicketPlanApproved` (Domain Event) — emitted when user approves the generated tickets

**External dependencies:** Reads DomainModel artifacts. Writes to Beads (infrastructure — issue tracker).

### Context: Ticket Freshness

**Responsibility:** Owns the ripple review lifecycle — flagging, context diff capture, review flow, and flag clearing.

**Key domain objects:**
- `RippleReview` (Aggregate) — tracks one ripple event (one ticket closure → N flagged tickets)
- `ContextDiff` (Value Object) — summary of what the closed ticket produced
- `FreshnessFlag` (Value Object) — the review_needed label + triggering ticket ID + context
- `TicketClosed` (Domain Event) — received from Beads; triggers the ripple
- `FlagCleared` (Domain Event) — emitted when a review is completed

**External dependencies:** Reads/writes ticket state via Beads (infrastructure).

### Context: Tool Translation

**Responsibility:** Owns the generation of tool-native configuration files from the domain model. One adapter per supported AI coding tool.

**Key domain objects:**
- `ToolConfig` (Aggregate) — the complete configuration for one AI tool
- `ToolAdapter` (Entity) — translates domain model into a specific tool's format
- `ToolConvention` (Value Object) — the known conventions for one tool (from Knowledge Base)
- `ConfigsGenerated` (Domain Event) — emitted when tool configs are ready for preview

**External dependencies:** Reads DomainModel artifacts. Reads ToolConventions from Knowledge Base. Writes to File Generation (generic).

### Context: Knowledge Base

**Responsibility:** Owns the curated reference material for DDD patterns, tool conventions, and coding standards. Provides RLM-addressable lookups.

**Key domain objects:**
- `KnowledgeEntry` (Entity) — a single addressable document (DDD pattern, tool convention, coding standard)
- `ToolVersion` (Value Object) — current + 3 previous major versions per tool
- `DriftReport` (Value Object) — detected divergence between documented conventions and actual tool behavior
- `KnowledgeUpdated` (Domain Event) — emitted when an entry is refreshed

**External dependencies:** None (self-contained reference store).

### Context: Bootstrap (Orchestration)

**Responsibility:** Owns the `alty init` flow — the orchestration of preview, confirmation, guided discovery, artifact generation, and file output. Coordinates the other contexts.

**Key domain objects:**
- `BootstrapSession` (Aggregate) — tracks the state of a `alty init` run
- `Preview` (Value Object) — what will be created/installed, shown before confirmation
- `Confirmation` (Value Object) — user's approval to proceed
- `GlobalSettingConflict` (Value Object) — detected conflict between global tool config and local settings
- `BootstrapCompleted` (Domain Event) — emitted when project is fully seeded

**External dependencies:** Orchestrates Guided Discovery, Domain Model, Architecture Testing, Ticket Pipeline, Tool Translation, File Generation.

### Context: Rescue (Gap Analysis)

**Responsibility:** Owns the `alty init --existing` flow — scanning existing projects, comparing against reference structure, generating gap reports and migration plans.

**Key domain objects:**
- `GapAnalysis` (Aggregate) — the scan results comparing existing project vs fully-seeded project
- `ProjectScan` (Value Object) — detected code, docs, configs, folder structure
- `Gap` (Entity) — a specific missing or conflicting element
- `MigrationPlan` (Entity) — ordered steps to bring existing project to seeded state
- `GapAnalysisCompleted` (Domain Event) — emitted when scan and report are ready

**External dependencies:** Reads existing project filesystem. Uses Bootstrap context for scaffolding.

## 5. Aggregate Design

### Aggregate: DiscoverySession (in Guided Discovery)

**Aggregate Root:** DiscoverySession

**Contains:**
- `Persona` (Value Object) — detected user persona
- `Register` (Value Object) — technical or non-technical
- `QuestionPhase` (Value Object) — current phase (Seed, Actors, Story, Events, Boundaries)
- `Question` (Entity) — current question with dual-register variants
- `Answer` (Value Object) — user's response per question
- `Playback` (Value Object) — summarization for confirmation

**Invariants (business rules this aggregate protects):**
1. Persona must be detected before questions start
2. Questions must proceed in phase order (Seed → Actors → Story → Events → Boundaries)
3. Playback must occur after every 3-4 questions
4. A question cannot be skipped without explicit user acknowledgment
5. Discovery cannot complete without at least the 5 minimum viable questions answered (Q1, Q3, Q4, Q9, Q10)

**Commands:**
- `detect_persona(choice)` — set persona and register
- `answer_question(question_id, response)` — record answer, advance to next question
- `confirm_playback(confirmed, corrections)` — confirm or correct the playback summary
- `skip_question(question_id, reason)` — skip with explicit acknowledgment
- `complete_discovery()` — finalize and emit DiscoveryCompleted

**Domain Events:**
- `PersonaDetected` — when persona is set
- `QuestionAnswered` — when user answers a question
- `PlaybackConfirmed` — when user confirms a playback
- `DiscoveryCompleted` — when all required questions answered and confirmed

### Aggregate: DomainModel (in Domain Model)

**Aggregate Root:** DomainModel

**Contains:**
- `DomainStory` (Entity) — one per business process
- `UbiquitousLanguage` (Entity) — glossary of domain terms
- `BoundedContextMap` (Entity) — contexts and relationships
- `AggregateDesign` (Entity) — per Core bounded context
- `SubdomainClassification` (Value Object) — per subdomain

**Invariants:**
1. Every term in UbiquitousLanguage must appear in at least one DomainStory
2. Every BoundedContext must have at least one SubdomainClassification
3. Every Core subdomain must have at least one AggregateDesign with invariants
4. BoundedContextMap relationships must be bidirectional (if A → B, B knows about A)
5. Ambiguous terms must be documented with per-context meanings

**Commands:**
- `add_domain_story(story)` — add a business process narrative
- `add_term(term, definition, context)` — add to ubiquitous language
- `add_bounded_context(context)` — add a bounded context
- `classify_subdomain(subdomain, classification, rationale)` — classify as Core/Supporting/Generic
- `design_aggregate(context, aggregate)` — add aggregate design for a Core context
- `finalize()` — validate all invariants and emit DomainModelGenerated

**Domain Events:**
- `DomainModelGenerated` — when model passes all validation

### Aggregate: FitnessTestSuite (in Architecture Testing)

**Aggregate Root:** FitnessTestSuite

**Contains:**
- `Contract` (Entity) — import-linter contracts
- `ArchRule` (Entity) — pytestarch rules
- `ContractStrictness` (Value Object) — per bounded context

**Invariants:**
1. Every bounded context must have at least one contract
2. Core subdomains must have all 4 contract types (layers, forbidden, independence, acyclic_siblings)
3. Supporting subdomains must have at least layers + forbidden
4. Generic subdomains must have at least one forbidden (ACL boundary)
5. No contract may reference a module outside its bounded context (except through defined relationships)

**Commands:**
- `generate_contracts(bounded_context_map, classifications)` — generate all contracts
- `preview()` — prepare human-readable preview
- `approve()` — mark as approved, emit FitnessTestsGenerated

**Domain Events:**
- `FitnessTestsGenerated` — when tests approved and ready for file output

### Aggregate: TicketPlan (in Ticket Pipeline)

**Aggregate Root:** TicketPlan

**Contains:**
- `GeneratedEpic` (Entity) — one per bounded context
- `GeneratedTicket` (Entity) — tasks within epics
- `DependencyOrder` (Value Object) — build sequence
- `TicketDetailLevel` (Value Object) — full/standard/stub per ticket

**Invariants:**
1. Every bounded context must produce exactly one epic
2. Foundation aggregates (no dependencies) must be ordered first
3. No circular dependencies in the generated ticket graph
4. Core subdomain tickets must have full AC, TDD phases, and SOLID mapping
5. No ticket may reference an aggregate from a different bounded context without going through a defined context relationship

**Commands:**
- `generate_plan(domain_model, classifications)` — generate epics and tickets
- `reorder(ticket_id, new_position)` — manually adjust order
- `promote_stub(ticket_id)` — expand a stub ticket to full detail
- `preview()` — prepare human-readable preview
- `approve(approved_ids, dismissed_ids)` — approve subset, emit TicketPlanApproved

**Domain Events:**
- `TicketPlanApproved` — when user approves the plan

### Aggregate: RippleReview (in Ticket Freshness)

**Aggregate Root:** RippleReview

**Contains:**
- `ContextDiff` (Value Object) — what the closed ticket produced
- `FreshnessFlag` (Value Object) — per flagged ticket

**Invariants:**
1. A ripple review must have a non-empty context diff (you can't flag without explaining what changed)
2. Only open tickets can be flagged (closed tickets are ignored)
3. A ticket can accumulate multiple flags from different closures (stacking)
4. Clearing a flag requires explicit review (no auto-clear)

**Commands:**
- `create_ripple(closed_ticket_id, context_diff)` — initiate ripple from a closure
- `flag_ticket(ticket_id)` — add review_needed flag with context
- `clear_flag(ticket_id, reviewer)` — mark as reviewed, update last_reviewed

**Domain Events:**
- `TicketFlagged` — when a ticket is marked review_needed
- `FlagCleared` — when a review is completed
