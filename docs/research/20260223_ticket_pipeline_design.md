# Domain Story to Ticket Pipeline Design

> **Spike:** k7m.11 -- Design the DDD artifact to beads ticket generation pipeline
> **Timebox:** 3 hours
> **Date:** 2026-02-23
> **Status:** Final

## Summary

This report designs the full pipeline from DDD artifacts to dependency-ordered beads
tickets. The pipeline reads a structured YAML intermediate representation (IR) extracted
from `docs/DDD.md`, applies complexity-budget-driven detail levels, topologically sorts
tickets by aggregate dependency, and writes them to beads with formal `bd dep add`
relationships. Human approval is required before any ticket is created.

## Research Questions Answered

1. What is the machine-readable input format for the pipeline?
2. How should tickets be ordered and sized?
3. What content goes into each detail level (FULL/STANDARD/STUB)?
4. How does the pipeline integrate with beads (CLI vs JSONL)?
5. How is PRD traceability generated for the user's project?
6. Should it be fully automated or human-in-the-loop?

## 1. DDD Artifact Input Format

### Problem

`docs/DDD.md` is a Markdown document designed for human readers. The ticket pipeline
needs structured, machine-readable data. Three options were considered:

| Option | Pros | Cons |
|--------|------|------|
| Parse Markdown directly | No extra file; single source of truth | Fragile regex parsing; breaks on format changes |
| YAML front matter in DDD.md | Colocated with human-readable doc | YAML block would be enormous; clutters the doc |
| **Separate YAML IR file** | Clean separation; easy to validate; generated by `alty generate artifacts` | Two files to keep in sync |

### Decision: Separate YAML IR at `.alty/domain-model.yaml`

The `alty generate artifacts` command (Domain Model context) already produces `docs/DDD.md`.
It will ALSO produce `.alty/domain-model.yaml` -- the machine-readable IR that all
downstream generators consume (fitness functions, ticket pipeline, tool translation).

This is consistent with the hexagonal architecture: `docs/DDD.md` is a human-readable
view (presentation), `.alty/domain-model.yaml` is the canonical domain model data
(application state).

### Key Design Decisions

1. **Subdomains are top-level.** The complexity budget lives at the subdomain level, not
   the bounded context level. A subdomain can contain multiple bounded contexts (e.g.,
   "Order Management" subdomain contains "Order Management" and "Fulfillment" contexts).

2. **Dependencies are on bounded contexts, not aggregates.** Cross-context dependencies
   define the epic-level build order. Intra-context aggregate dependencies (within one BC)
   define the ticket-level order within an epic.

3. **Aggregates only required for Core subdomains.** Supporting and Generic contexts have
   `aggregates: []` -- they get simpler tickets without aggregate-level detail.

4. **PRD capability IDs in domain stories.** This enables the traceability generation
   (section 6) by linking stories to PRD items.

## 2. Ticket Generation Algorithm

### 2.1 Overview

```
                    .alty/domain-model.yaml
                              |
                    [1. Parse & Validate]
                              |
                    [2. Classify Detail Levels]
                              |
                    [3. Generate Epics (1 per BC)]
                              |
                    [4. Generate Tickets per Epic]
                              |
                    [5. Topological Sort (cross-BC)]
                              |
                    [6. Set Intra-Epic Order]
                              |
                    [7. PRD Traceability Check]
                              |
                    [8. Preview for Approval]
                              |
                    [9. Write to Beads]
                              |
                    [10. Verify Dependency Graph]
```

### 2.2 Step Details

**Step 1: Parse & Validate**

- Load `.alty/domain-model.yaml`
- Validate against the schema (Pydantic model in domain layer)
- Enforce TicketPlan invariant #1: every BC produces exactly one epic
- Enforce invariant #5: cross-BC references must have a context_map entry

**Step 2: Classify Detail Levels**

- Read `subdomains[].treatment.ticket_detail` for each bounded context
- Map: `full` -> FULL template, `standard` -> STANDARD template, `stub` -> STUB template

**Step 3: Generate Epics (1 per BC)**

For each bounded context, generate one `GeneratedEpic`:

```
Epic: {BoundedContextName}
  - Subdomain: {subdomain.name} ({subdomain.classification})
  - Responsibility: {bc.responsibility}
  - Aggregates: {len(bc.aggregates)}
  - Detail level: {treatment.ticket_detail}
```

**Step 4: Generate Tickets per Epic**

Ticket generation rules differ by detail level:

| Detail Level | What generates tickets | Ticket count formula |
|-------------|----------------------|---------------------|
| **FULL** (Core) | 1 per aggregate + 1 per cross-BC integration + 1 epic setup | aggregates + integrations + 1 |
| **STANDARD** (Supporting) | 1 per major responsibility + 1 per integration | responsibilities + integrations |
| **STUB** (Generic) | 1 stub ("Integrate X, verify ACL boundary") | 1 |

For FULL detail (Core subdomains), each aggregate produces:

1. **Domain model ticket** -- Entity/VO/Aggregate definitions + invariant enforcement
2. **Command handler tickets** -- One per command (if >3 commands, group related ones)
3. **Integration ticket** -- Per cross-BC dependency (event handler or ACL adapter)

For STANDARD detail (Supporting subdomains):

1. **Service implementation ticket** -- All logic in one ticket
2. **Integration tickets** -- Per cross-BC dependency

For STUB detail (Generic subdomains):

1. **Integration stub** -- "Wrap X behind ACL, verify boundary test"

**Step 5: Topological Sort (Cross-BC Epic Order)**

Build a DAG from `context_map` relationships:

```python
# Pseudocode
graph = {}
for rel in context_map:
    graph[rel.downstream].add(rel.upstream)  # downstream depends on upstream

epic_order = topological_sort(graph)
# Result: foundation BCs first, dependent BCs later
```

**Invariant #2 enforcement:** Foundation aggregates (no dependencies) ordered first.
**Invariant #3 enforcement:** `topological_sort` raises `CycularDependencyError` if cycle detected.

Example for alty's own 9 BCs:

```
Tier 0 (foundation): Knowledge Base (no deps)
Tier 1:              Guided Discovery (no BC deps, but needs Knowledge Base for question content)
Tier 2:              Domain Model (depends on Guided Discovery via DiscoveryCompleted)
Tier 3:              Architecture Testing, Ticket Pipeline, Tool Translation (all depend on Domain Model)
Tier 4:              Ticket Freshness (depends on Ticket Pipeline via Beads)
Tier 5:              Bootstrap (orchestrates all), Rescue (orchestrates Bootstrap)
```

**Step 6: Set Intra-Epic Ticket Order**

Within each epic (single BC), order tickets by aggregate dependency:

1. Value objects and entities with no dependencies first
2. Aggregates that reference those entities next
3. Command handlers that use those aggregates
4. Integration tickets (event handlers, ACL adapters) last

This produces a deterministic, reproducible ordering within each epic.

**Step 7: PRD Traceability Check**

After generating all tickets, check coverage:

```python
# For each PRD capability ID referenced in domain_stories[].prd_capabilities
# Verify at least one generated ticket covers it
uncovered = prd_capabilities - ticket_covered_capabilities
if uncovered:
    report.add_warning(f"PRD capabilities not covered by any ticket: {uncovered}")
```

This implements the "coverage gap" report from the dogfooding insight.

**Step 8: Preview for Approval**

Display the full ticket plan in a structured preview:

```
Ticket Plan: example-project (14 tickets across 4 epics)

Epic 1: Order Management [CORE] (5 tickets)
  1.1 [FULL] Domain: Order aggregate (entities, VOs, invariants)
      Deps: none (foundation)
  1.2 [FULL] Command: place_order handler
      Deps: 1.1
  1.3 [FULL] Command: cancel_order handler
      Deps: 1.1
  1.4 [FULL] Integration: Fulfillment event subscription
      Deps: 1.1

Epic 2: Fulfillment [CORE] (3 tickets)
  2.1 [FULL] Domain: Shipment aggregate
      Deps: Epic 1 (Order Management)
  2.2 [FULL] Command: create_shipment handler
      Deps: 2.1
  2.3 [FULL] Integration: OrderPlaced event handler
      Deps: 2.1, Epic 1

Epic 3: Notifications [SUPPORTING] (2 tickets)
  3.1 [STANDARD] Service: Notification dispatch
      Deps: Epic 1, Epic 2
  3.2 [STANDARD] Integration: Event subscriptions
      Deps: 3.1

Epic 4: Payment Processing [GENERIC] (1 ticket)
  4.1 [STUB] Integrate Stripe behind ACL
      Deps: Epic 1

PRD Coverage: 2/2 capabilities covered (C1, C3)

[A]pprove all  [E]dit  [D]ismiss  [Q]uit
```

**Step 9: Write to Beads**

For each approved ticket, execute:

```bash
# 1. Create epic
bd create "Epic: Order Management" -t epic -p 1

# 2. Create child tickets
bd create "Domain: Order aggregate" -p 1 --parent <epic-id> --body-file /tmp/ticket_1_1.md

# 3. Set formal dependencies (CRITICAL)
bd dep add <ticket-1.2> <ticket-1.1>          # place_order depends on Order aggregate
bd dep add <ticket-2.1> <epic-1>              # Fulfillment depends on Order Management epic
bd dep add <ticket-2.3> <ticket-2.1>          # event handler depends on Shipment aggregate
bd dep add <ticket-2.3> <epic-1>              # event handler also depends on Order Management

# 4. Verify
bd blocked                                     # Should show downstream tickets
bd dep cycles                                  # Should show no cycles
```

**Step 10: Verify Dependency Graph**

After all tickets are written, run verification:

```python
# Verify invariant #3: no cycles
cycles = run_command("bd dep cycles")
assert cycles == "No cycles found"

# Verify invariant #2: foundation tickets are not blocked
ready = run_command("bd ready")
assert foundation_tickets are in ready_output

# Verify cross-BC deps are formal
blocked = run_command("bd blocked")
assert all downstream_tickets appear in blocked_output
```

### 2.3 Ticket Sizing Heuristics

Right-sizing avoids tickets that are too coarse (multi-day, unclear scope) or too fine
(trivial, overhead exceeds value).

| Classification | Sizing rule | Typical ticket size |
|---------------|-------------|-------------------|
| Core | 1 aggregate = 1-3 tickets (model + commands + integration) | 2-4 hours each |
| Supporting | 1 BC = 1-2 tickets (service + integration) | 2-4 hours each |
| Generic | 1 BC = 1 stub ticket | 1-2 hours |

**Coarseness guard:** If a Core aggregate has >5 commands, split into groups of 2-3
related commands per ticket.

**Fineness guard:** If a Core aggregate has only 1 command and no cross-BC integration,
merge the model and command into a single ticket.

## 4. Beads Integration Approach

### Options Considered

| Option | Pros | Cons |
|--------|------|------|
| **`bd create` + `bd dep add` via CLI** | Uses beads as intended; hooks fire; Dolt commits happen; labels work | Slower (one subprocess per ticket); error handling per call |
| Generate JSONL directly | Fastest; batch write | Bypasses Dolt; hooks don't fire; IDs must be manually computed; deps format undocumented |
| Hybrid: generate JSONL, then `bd import` | Batch + Dolt import | Still bypasses hooks; dep format in JSONL is internal |

### Decision: Use `bd create` + `bd dep add` via subprocess

Rationale:

1. **Beads is an external system with an ACL boundary.** The ticket pipeline's
   `BeadsWriterPort` should use the official CLI interface, not internal data formats.

2. **`bd create --body-file`** supports reading ticket description from a file, avoiding
   shell escaping issues with complex Markdown content.

3. **Formal dependencies are the critical requirement** (dogfooding insight). `bd dep add`
   is the only reliable way to create traversable dependencies. JSONL dependency format
   is internal and undocumented.

4. **Performance is acceptable.** A typical project generates 10-30 tickets. At ~100ms per
   `bd` invocation, the full pipeline takes 3-6 seconds. Preview + approval takes longer
   than writing.

### BeadsWriterPort Interface

```python
# src/application/ports/beads_writer_port.py
from __future__ import annotations
from typing import Protocol

class BeadsWriterPort(Protocol):
    def create_epic(
        self,

    def create_ticket(

        priority: int = 1,
        labels: list[str] | None = None,
    ) -> str:
        """Create a ticket as child of an epic. Returns the beads ticket ID."""
        ...

    def set_dependency(
        self,
        ticket_id: str,
        depends_on_id: str,
        dep_type: str = "blocks",
    ) -> None:
        """Set a formal blocking dependency between tickets."""
        ...

    def verify_no_cycles(self) -> list[str]:
        """Run bd dep cycles. Returns list of cycle descriptions (empty = good)."""
        ...

    def get_blocked(self) -> list[str]:
        """Run bd blocked. Returns list of blocked ticket IDs."""
        ...

    def get_ready(self) -> list[str]:
        """Run bd ready. Returns list of ready ticket IDs."""
        ...
```

### Infrastructure Implementation

```python
# src/infrastructure/external/beads_writer.py (sketch -- not production code)
import subprocess
import tempfile
from pathlib import Path

class BeadsCliWriter:
    """Implements BeadsWriterPort via bd CLI subprocess calls."""

    def create_epic(self, title: str, description: str, priority: int = 1) -> str:
        with tempfile.NamedTemporaryFile(mode="w", suffix=".md", delete=False) as f:
            f.write(description)
            body_file = f.name
        result = subprocess.run(
            ["bd", "create", f"Epic: {title}", "-t", "epic",
             "-p", str(priority), f"--body-file={body_file}"],
            capture_output=True, text=True, check=True,
        )
        return self._extract_id(result.stdout)

    def set_dependency(self, ticket_id: str, depends_on_id: str, dep_type: str = "blocks") -> None:
        subprocess.run(
            ["bd", "dep", "add", ticket_id, depends_on_id],
            capture_output=True, text=True, check=True,
        )
```

### Write Sequence

The write sequence must be ordered to ensure parent epics exist before child tickets,
and both tickets in a dependency exist before `bd dep add`:

```
Phase 1: Create all epics (no deps between epics yet)
Phase 2: Create all tickets under their parent epics
Phase 3: Set all intra-epic dependencies (ticket -> ticket within same epic)
Phase 4: Set all cross-epic dependencies (ticket -> epic, epic -> epic)
Phase 5: Verify (bd dep cycles, bd blocked, bd ready)
```

### Intra-BC Dependencies (Ticket Level)

Within a single bounded context, tickets are ordered by aggregate dependency:

1. **Foundation** -- Aggregates with no intra-BC dependencies (e.g., standalone VOs)
2. **Dependent** -- Aggregates that reference foundation aggregates
3. **Commands** -- Command handlers that operate on aggregates
4. **Integration** -- Event handlers and ACL adapters

This order is deterministic because it follows the DDD dependency direction:
Value Objects -> Entities -> Aggregates -> Commands -> Event Handlers.

### Circular Dependency Detection

The algorithm uses Kahn's algorithm (BFS-based topological sort), which naturally
detects cycles: if the sort completes without processing all nodes, the remaining
nodes form a cycle.

```python
def topological_sort_with_cycle_detection(graph: dict[str, set[str]]) -> list[list[str]]:
    """Kahn's algorithm with tier grouping.

    Returns list of tiers (each tier = nodes with no remaining deps).
    Raises CircularDependencyError if cycle detected.
    """
    in_degree = {node: 0 for node in graph}
    for node, deps in graph.items():
        for dep in deps:
            in_degree[dep] = in_degree.get(dep, 0)
            in_degree[node] += 1  # Conceptual -- actual impl counts incoming edges

    tiers = []
    queue = [n for n, d in in_degree.items() if d == 0]

    while queue:
        tier = sorted(queue)  # Sort for determinism
        tiers.append(tier)
        next_queue = []
        for node in tier:
            for dependent in reverse_graph[node]:
                in_degree[dependent] -= 1
                if in_degree[dependent] == 0:
                    next_queue.append(dependent)
        queue = next_queue

    processed = sum(len(t) for t in tiers)
    if processed < len(graph):
        remaining = [n for n in graph if n not in {x for t in tiers for x in t}]
        raise CircularDependencyError(f"Cycle detected involving: {remaining}")

    return tiers
```

### Coverage Gap Detection Algorithm

```python
def check_prd_coverage(
    domain_model: DomainModelYAML,
    ticket_plan: TicketPlan,
) -> list[CoverageGap]:
    """Check that every PRD capability referenced in domain stories
    is covered by at least one generated ticket."""

    # Collect all PRD capability IDs from domain stories
    all_capabilities: set[str] = set()
    for story in domain_model.domain_stories:
        all_capabilities.update(story.prd_capabilities)

    # Collect capabilities covered by tickets
    covered: set[str] = set()
    for epic in ticket_plan.epics:
        for ticket in epic.tickets:
            covered.update(ticket.prd_capabilities)

    # Report gaps
    gaps = []
    for cap_id in all_capabilities - covered:
        story = find_story_for_capability(domain_model, cap_id)
        gaps.append(CoverageGap(
            capability_id=cap_id,
            expected_contexts=story.bounded_contexts,
        ))

    return gaps
```

## 7. Human-in-the-Loop Design

### Decision: Preview + Approve (Never Fully Automated)

The pipeline generates a `TicketPlan` (domain model), previews it, and writes only
after explicit approval. This is consistent with all alty generation flows
(DDD.md Story 5, step 9-10).

### Approval Modes

| Mode | Description | Use case |
|------|-------------|----------|
| **Approve all** | Accept the entire plan as-is | Quick bootstrap, trust the pipeline |
| **Edit** | Modify specific tickets before creation | Adjust priority, add detail, remove tickets |
| **Selective approve** | Approve a subset, dismiss the rest | Phased rollout, only near-term tickets |
| **Dismiss all** | Reject the entire plan | Rethink DDD artifacts first |

### Edit Capabilities

Before approval, the user can:

1. **Reorder** -- Move a ticket to a different position (`TicketPlan.reorder()`)
2. **Promote stub** -- Expand a STUB ticket to STANDARD or FULL detail (`TicketPlan.promote_stub()`)
3. **Demote** -- Reduce a FULL ticket to STANDARD
4. **Remove** -- Drop a ticket from the plan
5. **Add** -- Manually add a ticket not generated by the pipeline

### MCP Variant

For MCP, the preview is returned as structured JSON (not terminal UI). The AI coding
tool presents it to the user and collects approval via its own UI.

```python
@mcp.tool()
def generate_tickets(project_dir: str, preview: bool = True) -> dict:
    """Generate dependency-ordered tickets from DDD artifacts."""
    plan = handler.generate_plan(project_dir)
    if preview:
        return {"status": "preview", "plan": plan.to_dict(), "stats": plan.stats()}
    # If preview=False, generate and write immediately (for non-interactive)
    handler.write_plan(plan)
    return {"status": "created", "ticket_count": plan.ticket_count}
```

## 9. Domain Model Design (TicketPlan Aggregate)

The TicketPlan aggregate from `docs/DDD.md` Section 5 maps to these domain classes:

```python
# Domain layer types (conceptual -- not production code)

@dataclass(frozen=True)
class TicketDetailLevel:
    """Value Object: full | standard | stub."""
    level: Literal["full", "standard", "stub"]

@dataclass(frozen=True)
class DependencyOrder:
    """Value Object: topologically sorted build sequence."""
    tiers: tuple[tuple[str, ...], ...]  # Immutable tiers
    epic_deps: tuple[tuple[str, str], ...]  # (downstream, upstream) pairs
    ticket_deps: tuple[tuple[str, str], ...]  # Intra-epic deps

@dataclass
class GeneratedTicket:
    """Entity: a task within a generated epic."""
    id: str  # Temporary ID before beads assignment
    title: str
    description: str  # Full Markdown content per template
    detail_level: TicketDetailLevel
    aggregate_name: str | None  # None for integration tickets
    bounded_context: str
    intra_deps: list[str]  # IDs within same epic
    cross_deps: list[str]  # Epic IDs from other BCs
    prd_capabilities: list[str]  # PRD capability IDs covered

@dataclass
class GeneratedEpic:
    """Entity: one epic per bounded context."""
    id: str  # Temporary ID
    title: str
    description: str
    bounded_context: str
    subdomain_classification: str
    tickets: list[GeneratedTicket]

@dataclass
class TicketPlan:
    """Aggregate Root: the complete generated ticket plan."""
    epics: list[GeneratedEpic]
    dependency_order: DependencyOrder
    prd_coverage: list[CoverageGap]

    # --- Invariant enforcement ---

    def validate(self) -> list[str]:
        """Validate all 5 invariants. Returns list of violation messages."""
        violations = []
        # Invariant 1: every BC -> exactly 1 epic
        # Invariant 2: foundation first
        # Invariant 3: no cycles
        # Invariant 4: Core tickets have full detail
        # Invariant 5: no cross-BC refs without relationship
        return violations

    # --- Commands ---

    def reorder(self, ticket_id: str, new_position: int) -> None: ...
```

## 11. Determinism and Reproducibility

The pipeline must produce identical output given identical input. This requires:

1. **Sorted iteration** -- When iterating bounded contexts, use alphabetical order
   (not dict insertion order). Within a tier, sort alphabetically.

2. **Stable ticket IDs** -- Temporary IDs are deterministic: `{epic_index}.{ticket_index}`
   based on sorted order. Beads assigns final IDs during creation.

3. **Deterministic Kahn's algorithm** -- When multiple nodes have in-degree 0, sort them
   alphabetically before adding to the tier.

4. **Template rendering** -- Use the same template engine with sorted keys.

## 12. Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| `bd create` subprocess fails mid-pipeline | High | Transactional write: track created IDs, rollback on failure (close + delete created tickets) |
| Domain model YAML out of sync with DDD.md | Medium | `alty generate artifacts` always regenerates both; add hash check |
| Ticket count too high for large projects | Medium | Cap at 50 tickets per pipeline run; warn and suggest phased generation |
| User modifies DDD.md manually, YAML stale | Medium | `alty generate tickets` checks `.alty/domain-model.yaml` freshness against `docs/DDD.md` mtime |
| Beads ID format changes | Low | BeadsWriterPort abstraction isolates format parsing |
| Large YAML file for complex domains | Low | YAML scales fine to hundreds of entries; no performance concern |

## Recommendation

**Build the pipeline in 3 phases:**

1. **Domain model YAML IR** (tickets 1-2) -- This schema is shared by fitness function
   generation (k7m.10) and tool translation. Build it first as shared infrastructure.

2. **Core algorithm** (tickets 3-5) -- Topological sort, template generation, TicketPlan
   aggregate with invariant enforcement. This is the novel capability.

3. **Integration** (tickets 6-11) -- Beads writer, preview, CLI/MCP commands, end-to-end
   tests.

The YAML IR schema (phase 1) should be designed together with the fitness function
spike (k7m.10) since both consume the same input. The `bounded_contexts` and
`subdomains` sections of the schema serve both pipelines.
