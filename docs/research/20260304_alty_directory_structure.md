# .alty/ Directory Structure and RLM Addressing Scheme

> **Spike:** alty-2j7.1 -- .alty/ directory structure and RLM addressing scheme
> **Timebox:** 4 hours
> **Research questions:**
> 1. What is the minimal `.alty/` directory structure that satisfies PRD section 5.1?
> 2. How should RLM addressing work for `alty kb <topic>` lookups?
> 3. What is the right format for `.alty/maintenance/doc-registry.toml`?
> 4. How do we seed `.alty/knowledge/` -- copy from a bundled template, or generate from domain model?

## 1. Existing Contracts

Before proposing any structure, the spike audited all existing code that interacts with
`.alty/` paths. Every design decision below preserves compatibility with these contracts.

### 1.1 KnowledgeLookupPort (application layer)

**File:** `src/application/ports/knowledge_lookup_port.py`

```python
class KnowledgeLookupPort(Protocol):
    def lookup(self, category: str, topic: str, version: str = "current") -> str: ...
    def list_tools(self) -> list[str]: ...
    def list_versions(self, tool: str) -> list[str]: ...
    def list_topics(self, category: str, tool: str | None = None) -> list[str]: ...
```

This port uses `(category, topic, version)` triples. The directory layout must support
deterministic path resolution from these three values.

### 1.2 KnowledgeReaderProtocol (application layer)

**File:** `src/application/queries/knowledge_lookup_handler.py`

Refined port that uses `KnowledgePath` value objects and returns `KnowledgeEntry`:

```python
class KnowledgeReaderProtocol(Protocol):
    def read_entry(self, path: KnowledgePath, version: str = "current") -> KnowledgeEntry: ...
    def list_topics(self, category: KnowledgeCategory, tool: str | None = None) -> tuple[str, ...]: ...
```

### 1.3 KnowledgePath value object (domain layer)

**File:** `src/domain/models/knowledge_entry.py`

- Valid categories: `ddd`, `tools`, `conventions`, `cross-tool`
- Path format: `{category}/{topic}` or `{category}/{tool}/{subtopic}`
- Validates no path traversal (`..`), non-empty, valid category prefix
- Properties: `.category`, `.topic`, `.tool`, `.subtopic`

### 1.4 FileKnowledgeReader (infrastructure layer)

**File:** `src/infrastructure/persistence/file_knowledge_reader.py`

Already implements O(1) path resolution:

```python
def _resolve_file_path(self, path: KnowledgePath, version: str) -> Path:
    if path.category == KnowledgeCategory.TOOLS:
        return self._root / "tools" / (tool or "") / version / f"{subtopic}.toml"
    if path.category == KnowledgeCategory.CROSS_TOOL:
        return self._root / "cross-tool" / f"{topic}.toml"
    return self._root / category_dir / f"{path.topic}.md"
```

### 1.5 KnowledgeDriftDetector (infrastructure layer)

**File:** `src/infrastructure/persistence/knowledge_drift_detector.py`

- Scans `.alty/knowledge/tools/` for drift signals
- Reads `_meta.toml` per tool directory
- Expects `[versions]` section with `tracked` list
- Reads `[_meta]` sections from individual TOML entries

### 1.6 DocHealthHandler (application layer)

**File:** `src/application/queries/doc_health_handler.py`

- Reads registry from `.alty/maintenance/doc-registry.toml`
- Expects `[[docs]]` TOML array entries with `path`, `owner`, `review_interval_days`
- Falls back to defaults (PRD.md, DDD.md, ARCHITECTURE.md) if registry is missing

### 1.7 BootstrapHandler (application layer)

**File:** `src/application/commands/bootstrap_handler.py`

- Plans to create `.alty/knowledge/_index.toml` as one of the bootstrapped files
- Uses `_PLANNED_FILES` tuple to build preview

### 1.8 RescueHandler (application layer)

**File:** `src/application/commands/rescue_handler.py`

- Checks for `.alty/knowledge/` path existence during gap analysis

## 3. `config.toml` Schema

### 3.1 MVP Schema

```toml
# .alty/config.toml
# Project-specific alty settings. Created by `alty init`.

[alty]
version = "0.1.0"                   # alty version that created this config
created_at = "2026-03-04"           # When this project was initialized

[project]
name = "my-project"                 # Project name (from README or user input)
language = "python"                 # Primary language
package_manager = "uv"              # Package manager

[tools]
# AI coding tools detected or selected during init
selected = ["claude-code"]          # Tools the user chose to configure
detected = ["claude-code"]          # Tools detected on the system

[knowledge]
# Knowledge base settings
seed_version = "1.0"                # Version of the seed content
```

### 3.2 Full Schema (Future)

```toml
[alty]
version = "0.1.0"
created_at = "2026-03-04"

[project]
name = "my-project"
language = "python"
package_manager = "uv"
root_package = "my_project"         # Python import root
src_path = "src"                    # Source directory

[tools]
selected = ["claude-code", "cursor"]
detected = ["claude-code", "cursor"]

[knowledge]
seed_version = "1.0"
auto_update = false                 # P2: automatic knowledge updates

[doc_health]
default_review_interval_days = 30   # Default if not specified per doc
knowledge_review_interval_days = 90 # Knowledge base review interval (PRD NFR)

[fitness]
enabled = true                      # Whether fitness functions are active
```

### 3.3 Design Decisions

- **TOML format** -- matches `pyproject.toml` convention, human-editable, stdlib `tomllib`
  for reading (Python 3.12+), `tomlkit` for writing (round-trip preservation)
- **Flat sections** -- no deep nesting, easy to parse with simple `tomllib.loads()`
- **No sensitive data** -- config contains project metadata only, safe to commit to git
- **`[knowledge].seed_version`** -- enables detection of outdated seed content when alty
  ships new knowledge entries. `alty doc-health --knowledge` can compare seed versions.

## 4. `doc-registry.toml` Schema

### 4.1 Schema

```toml
# .alty/maintenance/doc-registry.toml
# Tracks which project documents to monitor for health.
# Used by `alty doc-health` to check freshness, broken refs, and missing metadata.

# Each [[docs]] entry registers a document for tracking.
# Fields:
#   path                  - Relative path from project root (required)
#   owner                 - Person or team responsible (optional)
#   review_interval_days  - Max days between reviews before staleness (default: 30)

[[docs]]
path = "docs/PRD.md"
owner = "product"
review_interval_days = 30

[[docs]]
path = "docs/DDD.md"
owner = "architecture"
review_interval_days = 30

[[docs]]
path = "docs/ARCHITECTURE.md"
owner = "architecture"
review_interval_days = 30

[[docs]]
path = "README.md"
owner = "product"
review_interval_days = 90
```

### 4.2 Compatibility with Existing Code

The `FilesystemDocScanner.load_registry()` already parses this exact format:

```python
def _parse_toml_docs(content: str) -> list[dict[str, str | int]]:
    # Handles [[docs]] entries with path, owner, review_interval_days
```

And `DocHealthHandler.check()` already looks for the registry at:

```python
registry_path = project_dir / ".alty" / "maintenance" / "doc-registry.toml"
```

No code changes are needed to support this schema -- it is already implemented.

### 4.3 Fields from ARCHITECTURE.md Section 7.6

The architecture document specifies four metadata fields for drift detection in
knowledge entries. These apply to the `[_meta]` section of `.toml` knowledge files,
not to `doc-registry.toml`. The separation is intentional:

| Field | Location | Purpose |
|-------|----------|---------|
| `last_verified` | `[_meta]` in knowledge `.toml` | When the knowledge entry was last verified |
| `verified_against` | `[_meta]` in knowledge `.toml` | Tool version tested against |
| `confidence` | `[_meta]` in knowledge `.toml` | High/medium/low confidence level |
| `next_review_date` | `[_meta]` in knowledge `.toml` | When next review is due |
| `last_reviewed` | YAML frontmatter in project `.md` | When the doc was last reviewed |
| `review_interval_days` | `doc-registry.toml` | Max days between reviews |
| `owner` | `doc-registry.toml` | Who is responsible for the doc |

The `doc-registry.toml` tracks **project documents** (PRD.md, DDD.md, etc.).
The `[_meta]` sections track **knowledge entries** (tool conventions, DDD references).
These are checked by different commands:

- `alty doc-health` uses `doc-registry.toml` + frontmatter
- `alty doc-health --knowledge` uses `[_meta]` sections in `.alty/knowledge/` TOML files

### 4.4 Seed Content for `doc-registry.toml`

The registry should be seeded with the documents that `alty init` creates. The
default entries match what `DocHealthHandler` already falls back to:

```python
_DEFAULT_ENTRIES = (
    DocRegistryEntry(path="docs/PRD.md"),
    DocRegistryEntry(path="docs/DDD.md"),
    DocRegistryEntry(path="docs/ARCHITECTURE.md"),
)
```

The seed registry should add `README.md` (present in every project) and optionally
tool config files if the user selected tools during init.

## 5. RLM Addressing Convention

### 5.1 Address Format

RLM (Resource Locator Model) addresses follow this pattern:

```
{category}/{topic}                        # for ddd, conventions
{category}/{tool}/{subtopic}              # for tools
{category}/{topic}                        # for cross-tool
```

### 5.2 CLI Usage Mapping

```bash
alty kb ddd/tactical-patterns             # -> .alty/knowledge/ddd/tactical-patterns.md
alty kb ddd/strategic-patterns            # -> .alty/knowledge/ddd/strategic-patterns.md
alty kb conventions/tdd                   # -> .alty/knowledge/conventions/tdd.md
alty kb conventions/solid                 # -> .alty/knowledge/conventions/solid.md
alty kb tools/claude-code/agent-format    # -> .alty/knowledge/tools/claude-code/current/agent-format.toml
alty kb tools/cursor/rules-format         # -> .alty/knowledge/tools/cursor/current/rules-format.toml
alty kb cross-tool/concept-mapping        # -> .alty/knowledge/cross-tool/concept-mapping.toml
```

### 5.3 MCP Resource URI Mapping

```
alty://knowledge/ddd/tactical-patterns            # -> same file
alty://knowledge/tools/claude-code/agent-format    # -> same file (current version)
alty://knowledge/tools/claude-code/agent-format?version=v2.0  # -> v2.0/ version
alty://knowledge/conventions/tdd                   # -> same file
alty://knowledge/cross-tool/concept-mapping        # -> same file
```

### 5.4 Resolution Algorithm

The resolution algorithm is already implemented in `FileKnowledgeReader._resolve_file_path()`.
The mapping is O(1) -- direct path construction, no search, no index scan:

```python
def _resolve_file_path(self, path: KnowledgePath, version: str) -> Path:
    """Map a KnowledgePath to a concrete filesystem path.

    Resolution rules:
    1. TOOLS:       {root}/tools/{tool}/{version}/{subtopic}.toml
    2. CROSS_TOOL:  {root}/cross-tool/{topic}.toml
    3. DDD:         {root}/ddd/{topic}.md
    4. CONVENTIONS: {root}/conventions/{topic}.md
    """
    if path.category == KnowledgeCategory.TOOLS:
        return self._root / "tools" / path.tool / version / f"{path.subtopic}.toml"
    if path.category == KnowledgeCategory.CROSS_TOOL:
        return self._root / "cross-tool" / f"{path.topic}.toml"
    return self._root / path.category.value / f"{path.topic}.md"
```

### 5.5 Versioning for Tool Entries

- Default version is `"current"` (resolves to the `current/` directory)
- Explicit versions are `"v2.1"`, `"v2.0"`, etc. (resolve to `v2.1/`, `v2.0/` directories)
- `current/` is a physical directory containing the latest version files (not a symlink,
  to avoid cross-platform issues on Windows)
- `KnowledgeDriftDetector` reads `_meta.toml` for the `[versions].tracked` list to know
  which version directories to compare against

### 5.6 Listing Operations

```bash
alty kb --list                      # List all categories
alty kb --list ddd                  # List topics in ddd/
alty kb --list tools/claude-code    # List topics for claude-code (current version)
```

These map to `KnowledgeLookupHandler.list_categories()` and `.list_topics()`.

## 6. `_index.toml` Schema

### 6.1 Master Index

```toml
# .alty/knowledge/_index.toml
# Master index for the knowledge base. Used by alty kb --list.

[index]
schema_version = 1
generated_at = "2026-03-04"
seed_version = "1.0"

[categories]
ddd = { description = "DDD patterns and references", format = "markdown" }
conventions = { description = "Coding practice references (TDD, SOLID, quality gates)", format = "markdown" }
tools = { description = "AI coding tool conventions (versioned)", format = "toml" }
cross-tool = { description = "Cross-tool concept mappings", format = "toml" }
```

### 6.2 Purpose

The `_index.toml` serves three purposes:

1. **Discovery** -- `alty kb --list` reads this to show available categories
2. **Validation** -- Confirms the knowledge base was properly initialized
3. **Seed version tracking** -- `seed_version` enables upgrade detection

Note: The index is not needed for individual lookups (those are O(1) path construction).
It is only read for listing and health check operations.

## 7. Seed Content Strategy

### 7.1 Recommendation: Bundled Templates Copied at Init Time

**Decision:** Copy from bundled template files at `alty init` time. Do not generate
from the domain model.

**Rationale:**

1. **Knowledge content is generic, not project-specific.** DDD patterns, TDD references,
   and tool conventions are the same regardless of the user's domain. They do not depend
   on the domain model output.

2. **Domain model does not exist yet at seed time.** `alty init` creates `.alty/` before
   `alty guide` runs the DDD question flow. The domain-model.yaml is generated later,
   so knowledge entries cannot be derived from it.

3. **Copier template pattern aligns with existing architecture.** The alty project uses
   Copier for scaffolding (per prior spike research). Knowledge files are ideal template
   content -- they are static files that get copied into the project.

4. **Updates via `alty upgrade` in the future.** When alty ships new knowledge content,
   `alty upgrade` can compare `seed_version` in `config.toml` against the current alty
   version and offer to update knowledge files. This is a Copier `update` operation.

## 8. Port-to-Directory Mapping

This table shows how each port operation maps to the directory structure:

| Port Method | Input | Resolves To |
|-------------|-------|-------------|
| `KnowledgeReaderProtocol.read_entry(path="ddd/tactical-patterns")` | `category=ddd, topic=tactical-patterns` | `.alty/knowledge/ddd/tactical-patterns.md` |
| `KnowledgeReaderProtocol.read_entry(path="tools/claude-code/agent-format")` | `category=tools, tool=claude-code, subtopic=agent-format, version=current` | `.alty/knowledge/tools/claude-code/current/agent-format.toml` |
| `KnowledgeReaderProtocol.read_entry(path="tools/claude-code/agent-format", version="v2.0")` | Same, `version=v2.0` | `.alty/knowledge/tools/claude-code/v2.0/agent-format.toml` |
| `KnowledgeReaderProtocol.read_entry(path="cross-tool/concept-mapping")` | `category=cross-tool, topic=concept-mapping` | `.alty/knowledge/cross-tool/concept-mapping.toml` |
| `KnowledgeReaderProtocol.read_entry(path="conventions/tdd")` | `category=conventions, topic=tdd` | `.alty/knowledge/conventions/tdd.md` |
| `KnowledgeReaderProtocol.list_topics(category=DDD)` | `category=ddd` | Glob `.alty/knowledge/ddd/*.md`, return stems |
| `KnowledgeReaderProtocol.list_topics(category=TOOLS, tool="cursor")` | `category=tools, tool=cursor` | Glob `.alty/knowledge/tools/cursor/current/*.toml`, return stems |
| `DocHealthHandler.check()` | `project_dir` | Reads `.alty/maintenance/doc-registry.toml` |
| `KnowledgeDriftDetector.detect()` | `knowledge_dir` | Scans `.alty/knowledge/tools/*/current/*.toml` for `[_meta]` |

All mappings are deterministic O(1) path construction except `list_topics()` which
uses a single-directory glob (O(n) where n is topics in that category, typically < 10).

The proposed structure is fully compatible with all existing code:

- `FileKnowledgeReader._resolve_file_path()` already resolves paths correctly
- `KnowledgeDriftDetector` already scans `tools/*/current/*.toml`

### 9.2 Minor Additions Needed

| Addition | Description | Scope |
|----------|-------------|-------|
| Seed file bundle | Static knowledge files in `src/infrastructure/persistence/seed/` | New directory |
| `config.toml` renderer | Template rendering for `config.toml` with project vars | New file |

| Path | Git Status | Rationale |
|------|-----------|-----------|
| `.alty/config.toml` | Committed | Project settings, shared across team |

## 10. Risk Assessment

### 10.1 Maintenance Burden (MEDIUM)

Full structure: 4 tools x ~7 topics = ~28 TOML files to maintain.
**Mitigation:** Start with `current/` only (no version history directories). Add version
directories only when breaking changes occur. Seed files ship with alty and update via
`alty upgrade`.

### 10.2 Seed Content Staleness (LOW)

Knowledge files ship with alty releases. Between releases, tool conventions may change.
**Mitigation:** `alty doc-health --knowledge` checks `[_meta].next_review_date` and
`verified_against` fields. Users can manually update entries between alty releases.

### 10.3 Copier Template vs Plain Copy (LOW)

Using Copier for copying seed files adds a dependency for a simple file copy operation.
**Mitigation:** For MVP, use plain `shutil.copytree()` for the knowledge directory and
template rendering for `config.toml` only. Copier can be adopted later when template
update (`copier update`) becomes important.
