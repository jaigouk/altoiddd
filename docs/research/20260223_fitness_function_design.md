# Spike: Architecture Fitness Function Generation Design

> **Research Question:** How should vibe-seed map a bounded context map to executable
> architecture fitness function tests (import-linter TOML contracts + pytestarch test files)?
>
> **Timebox:** 4 hours
> **Status:** Complete

## 1. Executive Summary

**Recommendation:** Use a hybrid approach — generate **both** import-linter TOML contracts
(for CI-driven boundary enforcement) **and** pytestarch test files (for richer, code-based
rules). The generation pipeline is a pure domain model: bounded context map YAML is parsed
into a `FitnessTestSuite` aggregate, which emits `Contract` and `ArchRule` entities that
render to TOML and Python respectively.

**Key finding:** The bounded context map input schema needs only 5 fields per context
(name, classification, module_path, layers, relationships) to generate all contract types.
The FitnessTestSuite aggregate's 5 invariants from DDD.md can be enforced at generation
time — the pipeline validates completeness before emitting files.

**Risk:** pytestarch's `LayeredArchitecture` resolves module names relative to the scan
path. Module path conventions must match the user's actual project structure, which varies.
The `root_package` configuration in import-linter has the same concern. Both require the
bounded context map to declare the actual Python package path, not just the DDD name.

## 2. Bounded Context Map Input Schema

### Design Principle

The input schema captures the **minimum information needed** to generate all contract types.
It does not duplicate what import-linter or pytestarch already know (e.g., which files exist).
It captures **architectural intent** — what the DDD model says should be true.

### Schema Field Rationale

| Field | Purpose | Used By |
|-------|---------|---------|
| `project.root_package` | import-linter `root_package` | import-linter TOML |
| `project.src_path` | pytestarch `get_evaluable_architecture` scan path | pytestarch conftest |
| `module_path` | Python package name under root_package | Both |
| `classification` | Selects contract strictness (treatment level) | Both |
| `layers` | Which DDD layers exist in this context | layers contract, LayeredArchitecture |
| `aggregates` | Aggregate module names for independence contracts | independence contract (Core only) |
| `relationships` | Allowed cross-context imports | forbidden contract exclusions |
| `relationships.pattern` | Integration type (domain_event, query, acl) | Determines which module paths are allowed |
| `relationships.via` | The specific integration surface | ignore_imports in forbidden contracts |

### What the Schema Does NOT Capture

- **File locations** — import-linter and pytestarch discover these by scanning
- **Internal module structure** — only the top-level DDD layers are declared
- **Test locations** — convention-based (`tests/architecture/`)
- **External dependencies** — handled by `include_external_packages` flag per contract

## 3. DDD Concept to Architecture Test Mapping

### Mapping Table

| DDD Concept | Architecture Rule | import-linter Contract | pytestarch Rule |
|-------------|------------------|----------------------|-----------------|
| DDD layer hierarchy (domain < application < infrastructure) | Infrastructure can import application and domain; application can import domain; domain imports nothing above | `layers` contract with layers = [infrastructure, application, domain] | `LayeredArchitecture().layer("infrastructure")...layer("domain")` + `LayerRule` |
| Bounded context boundary | Context A cannot import Context B's internals (only via defined relationship) | `forbidden` contract: source = context_a.*, forbidden = context_b.* (with ignore for relationship.via) | `Rule().modules_that().are_sub_modules_of("A").should_not().import_modules_that().are_sub_modules_of("B")` |
| Domain layer purity | Domain layer has zero imports from infrastructure or external frameworks | `forbidden` contract: source = domain, forbidden = infrastructure + external packages | `Rule().modules_that().are_sub_modules_of("domain").should_not().import_modules_that().are_named("django", "sqlalchemy", ...)` |
| Aggregate isolation (Core) | Aggregates within same context do not import each other | `independence` contract: modules = [aggregate_a, aggregate_b] | `Rule().modules_that().are_named("agg_a").should_not().import_modules_that().are_named("agg_b")` |
| Acyclic sibling modules | No circular dependencies between sibling packages | `acyclic_siblings` contract: ancestors = [context.domain] | No direct equivalent — requires custom graph analysis |
| ACL boundary (Generic) | Application code cannot import generic context directly (must go through ACL/port) | `forbidden` contract: source = *.domain, forbidden = generic_context.* | `Rule().modules_that().are_sub_modules_of("domain").should_not().import_modules_that().are_sub_modules_of("cli")` |
| Port/adapter pattern | Infrastructure implements ports; domain depends on port interfaces only | `protected` contract: protected = ports.*, allowed_importers = infrastructure.* + domain.* | `Rule` checking that ports are only imported by expected layers |

### Contract Type Selection by Classification

| Classification | Required Contracts | import-linter Types | pytestarch Rules |
|---------------|-------------------|--------------------|--------------------|
| **Core** | All 4 + domain purity | `layers` + `forbidden` (cross-context) + `forbidden` (domain purity) + `independence` (aggregates) + `acyclic_siblings` | LayeredArchitecture + LayerRule + per-aggregate isolation + domain purity |
| **Supporting** | Layers + forbidden | `layers` + `forbidden` (cross-context) | LayeredArchitecture + LayerRule + cross-context boundary |
| **Generic** | ACL boundary only | `forbidden` (no direct import from domain/application) | Single Rule: domain/application cannot import generic directly |

# tests/architecture/test_guided_discovery.py
"""Architecture fitness functions for Guided Discovery bounded context (Core)."""
from __future__ import annotations

import pytest
from pytestarch import EvaluableArchitecture, LayeredArchitecture, LayerRule, Rule


@pytest.fixture(scope="session")
def guided_discovery_layers() -> LayeredArchitecture:
    return (
        LayeredArchitecture()
        .layer("infrastructure")
        .containing_modules(["myproject.guided_discovery.infrastructure"])
        .layer("application")
        .containing_modules(["myproject.guided_discovery.application"])
        .layer("domain")
        .containing_modules(["myproject.guided_discovery.domain"])
    )


class TestGuidedDiscoveryLayers:
    """Layer hierarchy: infrastructure -> application -> domain."""

    def test_domain_does_not_access_infrastructure(
        self,
        evaluable: EvaluableArchitecture,
        guided_discovery_layers: LayeredArchitecture,
    ) -> None:
        rule = (
            LayerRule()
            .based_on(guided_discovery_layers)
            .layers_that()
            .are_named("domain")
            .should_not()
            .access_layers_that()
            .are_named("infrastructure")
        )
        rule.assert_applies(evaluable)

    def test_domain_does_not_access_application(
        self,
        evaluable: EvaluableArchitecture,
        guided_discovery_layers: LayeredArchitecture,
    ) -> None:
        rule = (
            LayerRule()
            .based_on(guided_discovery_layers)
            .layers_that()
            .are_named("domain")
            .should_not()
            .access_layers_that()
            .are_named("application")
        )
        rule.assert_applies(evaluable)


class TestGuidedDiscoveryBoundary:
    """Context boundary: no imports from other bounded contexts."""

    def test_no_direct_import_of_domain_model(
        self, evaluable: EvaluableArchitecture

        rule = (
            Rule()
            .modules_that()
            .are_sub_modules_of("myproject.guided_discovery")
            .should_not()

            .import_modules_that()

## 4. Generation Pipeline Design

### Pipeline Architecture

```
                          Bounded Context Map YAML
                                    |
                                    v
                    ┌───────────────────────────────┐
                    │   BoundedContextMapParser      │  (Infrastructure: YAML reader)
                    │   Reads YAML, produces domain  │
                    │   value objects                 │
                    └───────────────┬───────────────┘
                                    |
                                    v
                    ┌───────────────────────────────┐
                    │   FitnessTestSuite Aggregate   │  (Domain: pure business logic)
                    │   generate_contracts() command  │
                    │   Validates 5 invariants        │
                    │   Emits Contract + ArchRule     │
                    │   entities                      │
                    └───────────────┬───────────────┘
                                    |
                          ┌─────────┴──────────┐
                          |                    |
                          v                    v
              ┌──────────────────┐  ┌──────────────────┐
              │ ImportLinter     │  │ PyTestArch       │
              │ ContractRenderer │  │ TestRenderer     │  (Infrastructure: file writers)
              │ → pyproject.toml │  │ → test_*.py      │
              │   TOML sections  │  │   files          │
              └──────────────────┘  └──────────────────┘
```

### Domain Model (Pure — No External Dependencies)

```
FitnessTestSuite (Aggregate Root)
├── project_name: str
├── root_package: str
├── contracts: list[Contract]
├── arch_rules: list[ArchRule]
└── invariant validation

Contract (Entity)
├── id: ContractId (value object)
├── name: str
├── contract_type: ContractType (enum: layers | forbidden | independence | acyclic_siblings | protected)
├── bounded_context: str
├── classification: Classification (enum: core | supporting | generic)
├── source_modules: list[str]
├── target_modules: list[str]         # layers list, forbidden list, independence list, etc.
├── ignore_imports: list[ImportPath]   # Allowed exceptions (defined relationships)
├── options: dict[str, Any]           # Type-specific options (ancestors for acyclic, etc.)

ArchRule (Entity)
├── id: ArchRuleId (value object)
├── name: str
├── rule_type: ArchRuleType (enum: layer_access | module_import | independence)
├── bounded_context: str
├── classification: Classification
├── subject_modules: list[str]         # modules_that().are_sub_modules_of(...)
├── forbidden_modules: list[str]       # should_not().import_modules_that()...
├── test_class: str                    # Generated pytest class name
├── test_method: str                   # Generated pytest method name

ContractStrictness (Value Object)
├── classification: Classification
├── required_contract_types: frozenset[ContractType]
├── @staticmethod from_classification(c) -> ContractStrictness

Classification (Enum)
├── CORE        → {layers, forbidden, independence, acyclic_siblings}
├── SUPPORTING  → {layers, forbidden}
├── GENERIC     → {forbidden}

BoundedContextSpec (Value Object — parsed from YAML)
├── name: str
├── module_path: str
├── classification: Classification
├── layers: list[str]
├── aggregates: list[str]
├── relationships: list[ContextRelationship]

ContextRelationship (Value Object)
├── target: str
├── direction: Direction (enum: upstream | downstream | both)
├── pattern: IntegrationPattern (enum: domain_event | query | acl)
├── via: str  # Module path of the integration surface
```

### Generation Algorithm

The `generate_contracts()` command on `FitnessTestSuite` follows this algorithm:

```
FOR each bounded_context in bounded_context_map:
    strictness = ContractStrictness.from_classification(context.classification)

    IF layers IN strictness.required_contract_types AND context.layers:
        EMIT layers Contract:
            layers = [root_package.context.module_path.layer for layer in context.layers]
            # Order: infrastructure > application > domain (top to bottom)

    IF forbidden IN strictness.required_contract_types:
        # Cross-context boundary
        other_contexts = all contexts EXCEPT current
        allowed_targets = [r.target for r in context.relationships]
        forbidden_targets = [c for c in other_contexts if c not in allowed_targets]
        ignore_imports = [
            f"{root_package}.{context.module_path}.{r.via} -> {root_package}.{target.module_path}"
            for r in context.relationships
        ]
        EMIT forbidden Contract for cross-context boundary

        # Domain purity (Core and Supporting only)
        IF classification != GENERIC:
            EMIT forbidden Contract:
                source = context.domain
                forbidden = context.infrastructure

    IF independence IN strictness.required_contract_types AND len(context.aggregates) > 1:
        EMIT independence Contract:
            modules = [root_package.context.domain.models.agg for agg in context.aggregates]

    IF acyclic_siblings IN strictness.required_contract_types:
        EMIT acyclic_siblings Contract:
            ancestors = [root_package.context.domain, root_package.context.application]

    # Generic contexts get a single forbidden contract from all domain layers
    IF classification == GENERIC:
        all_domain_modules = [root_package.ctx.domain for ctx in all non-generic contexts]
        EMIT forbidden Contract:
            source = all_domain_modules
            forbidden = [root_package.context.module_path]

VALIDATE 5 invariants:
    1. Every BC has >= 1 contract
    2. Core BCs have all 4 types
    3. Supporting BCs have layers + forbidden
    4. Generic BCs have >= 1 forbidden
    5. No contract references module outside BC except via defined relationship
```

## 5. ADR: Hybrid Approach (import-linter + pytestarch)

### Status: Accepted

### Context

Two tools exist for Python architecture testing with complementary strengths:

| Aspect | import-linter | pytestarch |
|--------|--------------|------------|
| Configuration | TOML in pyproject.toml | Python test code |
| Execution | CLI (`lint-imports`) | pytest (`pytest tests/architecture/`) |
| CI integration | Standalone step or pytest wrapper | Part of pytest suite |
| Contract expressiveness | 6 built-in types + custom | Fluent Rule API + LayerRule |
| Acyclic siblings | Native support | No direct equivalent |
| Layer architecture | Native contract type | LayeredArchitecture + LayerRule class |
| Generated output | Declarative config | Imperative test code |
| Human readability | High (TOML is scannable) | Medium (Python code) |
| Maintenance burden | Edit TOML | Edit Python |
| Error messages | Lists violating import chains | Lists violating imports |

### Decision

Generate **both** import-linter TOML contracts and pytestarch test files.

### Rationale

1. **import-linter is the CI gate.** It runs as `lint-imports` or integrates into existing
   quality gate checks. TOML contracts are easy to read, review, and modify. The `layers`,
   `independence`, and `acyclic_siblings` contract types have no pytestarch equivalent.

2. **pytestarch provides richer, context-specific rules.** Domain purity checks (no external
   framework imports), cross-context boundary tests with relationship exceptions, and
   per-aggregate isolation are more naturally expressed as Python test code. Tests run with
   `pytest` alongside unit tests.

3. **Complementary coverage eliminates gaps.**
   - import-linter handles: layer hierarchy, acyclic siblings, aggregate independence
   - pytestarch handles: cross-context boundaries with exceptions, domain purity, custom rules
   - Overlap in layer checking is intentional — defense in depth

4. **Both are actively maintained with permissive licenses.**
   - import-linter v2.10 (Feb 2026), BSD-2
   - pytestarch v4.0.1 (Aug 2025), Apache-2.0

### Consequences

- Users get two `pyproject.toml` dependencies: `import-linter` and `pytestarch`
- Quality gate command becomes: `uv run lint-imports && uv run pytest tests/architecture/`
  (or unified via `vs check --fitness`)
- Generated projects have both `pyproject.toml` TOML sections and `tests/architecture/*.py` files
- Maintenance: if a user modifies their bounded context map, they re-run `vs generate fitness`
  to regenerate both outputs

### Alternatives Considered

| Alternative | Why Rejected |
|-------------|--------------|
| import-linter only | Cannot express cross-context boundary rules with relationship exceptions; no pytest integration |
| pytestarch only | No acyclic_siblings or independence contract equivalent; less scannable output |
| deply only | Not a pytest plugin; no `pyproject.toml` integration; Python 3.12 support unclear |
| grimp (DIY) | Too low-level; would duplicate import-linter's contract logic |

## 7. FitnessTestSuite Invariant Enforcement

The 5 invariants from DDD.md Section 5 are enforced at **generation time** (before any
file is written), not at test runtime:

| # | Invariant | Enforcement Point |
|---|-----------|-------------------|
| 1 | Every BC must have >= 1 contract | After generation loop: assert len(contracts_for_bc) >= 1 for all BCs |
| 2 | Core subdomains must have all 4 types | After generation loop: assert {layers, forbidden, independence, acyclic_siblings} <= contract_types_for_bc |
| 3 | Supporting must have layers + forbidden | After generation loop: assert {layers, forbidden} <= contract_types_for_bc |
| 4 | Generic must have >= 1 forbidden | After generation loop: assert ContractType.FORBIDDEN in contract_types_for_bc |
| 5 | No contract references module outside BC except via defined relationship | During forbidden contract generation: verify all ignore_imports reference a defined relationship |

If any invariant is violated, `generate_contracts()` raises a domain error and no files
are emitted. This is a **fail-fast** design — the user sees what's wrong with their
bounded context map before any output is generated.

## 8. Integration with vs CLI and Quality Gates

### CLI Command

```
vs generate fitness [--preview] [--context <name>]
```

- `--preview`: Show what would be generated without writing files (default behavior per
  DDD.md Domain Story 4, step 6)
- `--context <name>`: Generate for a single bounded context (useful during development)
- Without flags: generate all, show preview, wait for approval

### Quality Gate Integration

From the CLI design doc (`docs/research/20260222_cli_mcp_design.md`), the `vs check`
command already has a `--fitness` flag:

```
vs check --fitness    # Runs: lint-imports && pytest tests/architecture/
```

This maps to the `QualityGatePort` in the application layer.

### pyproject.toml Integration

Generated import-linter contracts are appended to the user's `pyproject.toml` under
`[tool.importlinter]`. The renderer must:

1. Read existing pyproject.toml (if any)
2. Check for existing `[tool.importlinter]` section
3. If exists: merge (replace contracts with same name, add new ones)
4. If not exists: add `[tool.importlinter]` section with `root_package`
5. Write back using `tomlkit` (preserves formatting and comments)

**Dependency:** `tomlkit` (MIT license, 688 stars, actively maintained) for
round-trip TOML editing. Alternative: `tomli-w` (write-only, lighter) but does not
preserve existing formatting.

## 9. Open Questions and Risks

### Resolved

| Question | Resolution |
|----------|-----------|
| import-linter vs pytestarch vs both? | Both (hybrid) — see ADR in Section 5 |
| How to handle cross-context relationships? | `ignore_imports` in forbidden contracts; relationship.via field in schema |
| What drives contract strictness? | Classification (Core/Supporting/Generic) from complexity budget |

### Open

| Question | Impact | Resolution Path |
|----------|--------|-----------------|
| Should Bootstrap and Rescue contexts get fitness functions? | Bootstrap is an orchestrator (no domain layer); Rescue is P1 (deferred). Probably skip both. | Decide during k7m.7 architecture review |
| How to handle `include_external_packages` for domain purity? | import-linter needs this flag to check external imports (django, sqlalchemy). Should we default to True for Core domain purity contracts? | Implementation decision — likely yes |
| pytestarch module resolution from project root | pytestarch resolves modules relative to the scan path. If `src_path = "src"`, modules are `src.myproject.domain` not `myproject.domain`. Need to verify correct path convention. | Test during implementation with a real project structure |
| TOML rendering library | `tomlkit` vs `tomli-w` for pyproject.toml editing | Implementation decision — tomlkit preferred for round-trip preservation |
| How to regenerate without losing manual edits? | User may add custom contracts. Regeneration should preserve user-added contracts (marked with a comment). | Design a `# vibe-seed:generated` marker convention |

### Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| pytestarch drops Python 3.12+ support or goes unmaintained | Medium | import-linter covers all critical contracts; pytestarch tests are supplementary. Monitor PyPI releases. |
| Module path mismatch between YAML schema and actual project | High | Validate module paths during generation by checking filesystem existence (infrastructure concern, not domain) |
| 28 contracts + 35 tests may be overwhelming for small projects | Medium | `--context` flag for incremental generation; generated files have clear comments; `vs check --fitness` hides complexity |
| import-linter TOML becomes large in pyproject.toml | Low | Consider separate `.importlinter` config file (import-linter supports this). Or generate to `importlinter.toml` if pyproject.toml is too crowded. |
